## 1. I/O 란?

그 전에 I/O가 무엇인지부터 정리하고 넘어가겠습니다.

I/O는 **Input/Output**, 즉 **입출력**을 의미합니다. 컴퓨터가 데이터를 주고받는 모든 작업이죠.

구체적으로 어떤 것들이 있을까요? 디스크에서 파일을 읽고 쓰는 **파일 I/O**, 네트워크를 통해 데이터를 송수신하는 **네트워크 I/O**, 사용자로부터 키보드나 마우스 입력을 받는 것, 이 모든 것이 I/O 작업입니다.

> **"Everything is a File"**
> 

운영체제나 파일 시스템 강의에서 한 번쯤 들어보셨을 겁니다. Unix/Linux에서는 소켓도 파일, 디바이스도 파일, 파이프도 파일로 추상화됩니다. 모든 I/O 자원을 파일 디스크립터(File Descriptor)라는 하나의 인터페이스로 다룰 수 있습니다.

그래서 **I/O 작업**은 단순히 서버 내에서 일어나는 읽기/쓰기뿐만 아니라, **네트워크 통신, 장치 제어 등 폭넓은 의미**를 가집니다. 그리고 이 모든 것이 결국 **파일 디스크립터를 통한 I/O**로 귀결됩니다.

## 2. I/O 의 4가지 개념

이렇듯 I/O를 효율적으로 처리하는것이 중요하다는 것을 알게 되었으니, I/O에서 자주 통용되는 핵심 개념을 이해해보아야 합니다.

발표 이전까지, 너무나도 많이 들어왔지만 정작 설명하라고 하면 설명하지 못할 개념들을 이번 기회에 정리하게 되어서 좋았습니다!

### 2.1 Blocking / Non-Blocking

먼저, Blocking과 Non-Blocking입니다. 정의부터 살펴보겠습니다.

<aside>

- Blocking
    
    요청한 작업이 모두 완료될 때까지 기다렸다가, 완료될 때 응답과 결과를 반환받는다. 요청 작업 결과를 기다린다.
    
- Non-Blocking
    
    작업 요청 이후 결과는 필요할 때 전달받는다. 요청 작업 결과를 기다리지 않는다.
    
    중간중간 필요에 따라 작업 상태 확인이 가능하다.
    
</aside>

**Blocking**의 경우

요청한 작업이 완료될때까지 기다려야 합니다. 그렇기 때문에 프로세스는 대기 시간동안 아무것도 하지 못한다는 단점이 있습니다.

**Non-Blocking**의 경우

I/O 작업 요청후 기다리지 않고 즉시 제어권을 돌려받습니다.

작업 결과는 나중에 필요할 때 전달받습니다.

저는 이 두 개념을 각각 ‘제어권을 언제 돌려받는지’ ,  ‘응답을 어떤 타이밍에 받는지’ 의 개념으로 이해했습니다.

### 2.2 Synchronous / Asynchronous

다음으로, Synchronous 와 Asynchronous 의 개념을 알아보겠습니다.

<aside>

- Synchronous
    
    모든 I/O 요청-응답 작업이 **일련의 순서**를 따릅니다.
    
    즉, 작업 순서가 보장됨을 의미합니다.
    
    작업 완료 시점을 프로세스가 판단하고, 다음 작업 요청 시점을 결정하게 합니다.
    
- Asynchronous
    
    커널에 I/O 작업을 요청해두고 다른 작업 처리가 가능합니다.
    
    **그러나 작업 순서가 보장되지 않습니다.**
    
    작업이 완료되면, 커널에서 프로세스에게 통보합니다.
    
</aside>

**Synchronous** 의 경우

![image.png](image.png)

Synchronous 는 호출한 쪽에서 작업 완료를 직접 확인합니다.

그렇기 때문에 작업 완료 여부에 따라서 다음 작업을 배정하기 때문에, 작업 순서가 보장된다는 것입니다.

**Asynchronous** 의 경우

![image.png](image%201.png)

Asynchronous 는 반면 호출한 쪽에서 작업에 대한 완료에 대해 전혀 신경쓰지 않습니다.

대신 작업 완료시 Event나 callback을 기반으로 해당 작업이 완료됨을 통지받습니다.

저는 이 두 개념을 각각 ‘작업 완료를 확인하는것이 누구인지’ ,  ‘작업의 순서가 보장되는지’ 의 개념으로 이해했습니다.

정리하자면, **Blocking/Non-Blocking**은 **제어권 반환 시점**에 관한 것이고, **Sync/Async**는 **완료 확인 주체**에 관한 것입니다.

그리고 이 둘은 독립적인 개념이기 때문에 **4가지 조합**이 가능합니다.

OS에 따라서 세부적인 차이가 존재하지만, 여기서는 Linux 기반의 I/O 모델을 살펴보겠습니다.

![image.png](image%202.png)

- **Blocking + Sync**
    
    ![image.png](image%203.png)
    
    가장 일반적인 Blocking I/O 모델입니다.. 완료까지 대기하고, 직접 결과를 확인합니다.
    
    프로세스가 커널에게 I/O를 요청하는 system call을 호출한 뒤, 커널이 작업 결과를 반환하기까지 중단하고 대기합니다. 커널의 응답과 데이터가 반환되면 해당 시점에서 프로세스가 unblocking되어 반환 데이터를 처리합니다.
    
- **Non-Blocking + Sync**
    
    ![image.png](image%204.png)
    
    대기 시간동안 다른 작업이 가능하나 결국 프로세스는 계속 상태를 체크해야 합니다.
    
    이를 Busy-Wait이라고 표현합니다.
    
    이렇게 반복적으로 데이터가 준비되었는지 확인하는 Polling 작업이 필요하며, CPU의 Context-switching을 빈번하게 일으킵니다.
    
    이러한 Polling의 주기를 적절하게 설정하지 않는다면, 주기가 너무 길어 데이터가 준비되어도 후속 처리가 늦어지거나, 주기가 너무 짧아 의미 없는 요청을 반복하게 되어 작업의 지연을 초래할 수 있습니다.
    

위의 2가지 Sync 방식은 결국 요청한 순서대로 작업을 완료시킵니다.

이는 직관적이지만, **2개 이상의 파일**을 동시에 처리하기에는 동기화와 같은 복잡한 이슈가 발생하게 됩니다.

그 때문에, 이를 해결하기 위한 대안으로 **I/O Multiplexing** 을 사용합니다. 이에 대해서는 잠시 뒤에 설명합니다.

- **Blocking + Async**
    
    ![image.png](image%205.png)
    
    `select()`, `epoll()` 같은 시스템 콜을 호출하면 **여러 FileDescriptor를 커널이 비동기로 감시**하고, 프로세스는 그동안 **Blocking 대기**합니다.
    
    **준비된 FD가 생기면** 커널이 알려주고, 프로세스는 해당 FD만 처리합니다.
    
    - **Blocking -** `epoll_wait()`에서 대기
    - **Async -**  커널이 알아서 감시하다가 준비되면 알려줌
    - **장점**
        
        Polling 없이 효율적, 하나의 스레드로 수천 개 연결 관리 가능
        
- **Non-Blocking + Async**
    
    ![image.png](image%206.png)
    
    이는 **Asynchronous Non-Blocking I/O** 로 불리며, 줄여서 AIO라고 합니다.
    
    이 환경에서 사용자 프로세스는 시스템 콜 호출 이후 처리에 신경쓰지 않고 있다가, 작업 완료시 커널로부터 signal, thread 기반 callback 등으로 결과를 event처럼 전달받아 처리합니다.
    
    그렇기 때문에 사용자 프로세스는 I/O와 독립적인 다른 작업이 가능합니다
    

## 3. I/O Multiplexing

그렇다면 I/O Multiplexing이 무엇인지 알아보기 전에, C10K 문제가 무엇인지 잠시 알아보고 가겠습니다.

> C10K 문제란? 단일 서버가 동시에 10,000명(Concurrent 10K connections) 이상의 클라이언트와 연결을 유지하며 통신할 수 있도록 **네트워크 소켓 처리 성능을 최적화하는 문제**를 말한다.
> 

이러한 문제를 해결하기 위한 기법 중 멀티 프로세싱 → 멀티 스레딩 다음으로 제시된 기법이 Multiplexing 기법이며 이는 근본적으로 **하나, 혹은 소수의 스레드가 여러 I/O를 동시에 관리할 수 있게 하는 기법입니다.**

적은 수의 스레드를 사용하여 얻을 수 있는 이득은 다음과 같습니다.

1. **메모리 낭비의 감소**
2. **Context-Switching 오버헤드 감소**
3. **동기화 문제 해결 가능**

이제 실제로 Multiplexing이 어떻게 동작하는지 확인해보겠습니다.

### 3.1 Multiplexing의 동작정리

**용어정리**

본격적으로 들어가기 전에, 헷갈리기 쉬운 용어들을 정리하겠습니다.

**Server Process (서버 프로세스)**

- Redis, Nginx 같은 서버 애플리케이션
- User Space에서 실행됨

**Kernel (커널)**

- OS의 핵심 부분
- Kernel Space에서 실행되며 하드웨어와 직접 통신
- I/O 작업을 실제로 수행

**Client (클라이언트)**

- 서버에 접속하는 외부 프로그램
- 예: redis-cli, 웹 브라우저

**Socket (소켓)**

- 네트워크 통신의 끝점
- 서버 프로세스가 클라이언트와 통신하기 위한 **파일 디스크립터(FD)**
- **클라이언트 1명 = 소켓 1개**

**Thread (스레드)**

- 프로세스 내에서 실행되는 실행 단위
- Redis는 **1개의 메인 스레드**로 동작

---

기존의 전통적인 멀티스레딩은 다음과 같이 동작합니다.

<aside>

**기존의 멀티스레딩 방식 서버 프로세스**

Thread 1 → Socket A → read() → Blocking 대기

Thread 2 → Socket B → read() → Blocking 대기

Thread 3 → Socket C → read() → Blocking 대기

</aside>

각 스레드는 자신이 담당하는 소켓에서 데이터가 도착하기까지 Blocking 상태로 다기합니다.

```sql
// Thread 1이 Socket A 담당
while(1) {
    read(socket_A, buffer, size);
    // 데이터가 올 때까지 무한정 대기
}
```

그렇기 때문의 위의 C10K 문제 해결을 위해서 10000개의 연결을 위해 발생하는 메모리와 문맥교환 비용이 커지게 됩니다.

---

반면, Multiplexing I/O모델은 select, poll, epoll 같은 시스템 콜을 통해 여러 소켓을 동시에 감시하고, 실제로 읽을 준비가 된 소켓에 대해서만 I/O작업을 수행합니다.

<aside>

**Redis 와 같은 Multiplexing I/O 서버 프로세스**
1개 메인 Thread ← 메인 스레드가 모든 클라이언트 소켓을 관리함.

↓

epoll_wait([Socket A, B, C, ...]) ← 해당 시점에서 스레드는 Blocking 상태로 대기

↓

커널 : "Socket B, D가 준비됨!"

↓

read(Socket B) → 즉시 데이터 읽기 (Non-Blocking처럼 동작)

read(Socket D) → 즉시 데이터 읽기

↓

다시 epoll_wait()

</aside>

- **epoll이란?**
    
    Linux에서 사용하는 I/O Multiplexing 시스템 콜중 하나입니다.
    
    앞서 말한 select()나, 이전에 사용하던 poll()의 경우 전체 파일 디스크립터(소켓을 관리하는)들을 순차적으로 관리했습니다.
    
    즉 모든 소켓을 매번 1회의 순회로 탐색하여 어떤 소켓이 준비되었는지 일일이 확인하는데 O(n)만큼 시간이 소요되었습니다.
    
    epoll의 경우 소켓이 준비 되었을 경우에만 리스트에 등록합니다.
    
    실제 네트워크 서버에서는 소켓 활성화 비율이 낮기 때문에 select() 보다 더 효율적으로 사용 가능한 시스템 콜입니다.
    

```sql
//Main Thread 1개가 모든 소켓 관리
while(1) {
    // 읽기가 준비된 소켓만을 사용
    int n = epoll_wait(epfd, events, ...);
    
    // 준비된 소켓만 처리
    for(int i = 0; i < n; i++) {
        int fd = events[i].data.fd;
        read(fd, buffer, size);  // 데이터의 존재를 확인 후 읽기
    }
}
```

커널은 **이벤트가 발생한 소켓만을 선별**해 애플리케이션에 알려주며, 애플리케이션은 그 후에 해당 소켓에 대해 실제 I/O작업을 수행하게 됩니다.

이로 인해 **입출력 함수 자체는 여전히 블로킹**이지만, **실제로 데이터가 준비된 소켓에 대해서만 호출**되므로, 스레드가 소켓마다 블로킹되어야 했던 구조적 제약을 완화하고, 이벤트 기반으로 필요한 순간에만 워커 스레드를 투입할 수 있도록 했습니다.

이러한 구조를 통해 하나의 스레드에서 수천 개의 소켓 상태를 동시에 감시할 수 있게 해주어, 적은 수의 스레드로도 높은 동시성을 달성할 수 있도록 하고 있습니다!

### 3.2 Multiplexing의 사용
